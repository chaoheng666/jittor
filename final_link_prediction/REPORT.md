# 比赛方案报告：从零理解我们的思路、模型和最终结果

## 1. 一句话总结

我们做的是一个图上的链接预测任务：每条测试样本给出 100 个候选目标，模型要把最可能的目标排到前面。最终方案没有把稳定基线推翻重做，而是在稳定基线之上加入少量上下文增强排序模型信号。已知最佳公开分是 `1.2829`，对应提交包是 `result_final_blend_0p10.zip`。

## 2. 任务是什么

比赛给了两类数据：`dataset1` 和 `dataset2`。每一条测试数据可以理解成：

```text
给定一个源对象 src、一个时间 time、100 个候选目标 dst，要求给这 100 个候选目标各打一个概率分数。
```

提交文件需要包含 `dataset1.csv` 和 `dataset2.csv`。每一行的 100 个概率加起来应该接近 1。真实答案排得越靠前，分数越高。

## 3. 重要术语

- 图：由点和边组成的数据结构。这里可以把 `src` 和 `dst` 看作点，把一次历史连接看作边。
- 图学习：从点和边的历史关系中学习规律，用来预测未来可能出现的新关系。
- 候选集：每条测试数据给出的 100 个可能答案。
- 排名模型：模型不只判断“是不是”，而是给 100 个候选排序。
- 特征：把原始数据转成模型可计算的数字，例如目标是否热门、源对象最近连接过什么、候选是否与历史行为相似。
- embedding：把编号压缩成连续向量，用来表达对象之间的相似关系。
- logits：模型输出的原始分数，还不是概率。
- softmax：把 logits 转成概率的函数，转换后一行概率总和为 1。
- MRR：平均倒数排名。真实答案排第 1 得 1 分，排第 2 得 1/2，排第 10 得 1/10。
- ensemble：模型融合，把多个模型的分数合起来，通常比单个模型更稳。
- blend：按比例混合两个模型。例如 `0p10` 表示加入 10% 的增强模型信号。
- top1 change：第一名候选被改变的比例。比例越高，模型越激进，线上风险通常越大。

## 4. 为什么不直接用一个更大的深度模型

测试时每行只有 100 个候选，模型真正要做的是候选内排序；同时线上测试分布不一定和本地验证完全一致。一个新模型如果改变了大量第一名候选，线上可能涨，也可能跌。最后一次提交机会不能押太激进的方案。

所以最终策略是：保留稳定基线作为主体，让上下文增强排序模型只提供补充信号，用 `0.10` 融合比例控制风险。

## 5. 方案组件

### 稳定基线

稳定基线使用规则特征和 MLP 模型。规则特征包括目标热度、源对象最近连接历史、候选是否在历史中出现、SVD/embedding 相似度、时间和转移关系。MLP 是一个小型神经网络，用这些特征给候选打分。它的优势是稳定，缺点是提升空间有限。

### 困难候选构造

普通训练样本里很多负例太容易分辨，模型学不到细节。最终方案会把稳定基线容易打高分的错误候选放进训练集，并混入近期热门目标、历史转移相关目标和低热度目标。每条训练样本仍保持 100 个候选，贴近真实测试格式。

### 上下文增强排序模型

上下文增强排序模型在原有图特征之外，加入序列和受众信息：源对象最近行为序列、目标对象最近吸引过哪些源对象、源对象和目标受众的相似度、历史序列里的相似度最大值/平均值/最后一次相似度。

模型训练 3 个不同随机种子：`3101, 3102, 3103`。预测时取平均，降低随机波动。

## 6. 最终融合策略

最终没有直接使用纯增强模型，而是：

```text
最终 dataset2 分数 = 90% 稳定基线 + 10% 上下文增强排序模型
```

两个模型的 logits 先做逐行 z-score 标准化，再按比例相加，最后用 softmax 转成提交概率。这个策略的核心是风险控制：稳定基线决定大部分排序，增强模型只修正一小部分候选。

## 7. 已知结果

- 文件：`result_final_blend_0p10.zip`
- 公开分：`1.2829`
- dataset1：复用稳定基线中表现最好的结果
- dataset2：稳定基线和上下文增强排序模型按 90/10 融合

也可以重新打包其他融合比例，例如 `0.02`、`0.05`、`0.20`、`0.35`、`1.00`。但比例越大越激进，最后一次提交更建议以 `0.10` 为主。

## 8. 重新生成要多久

如果 baseline artifacts 和最终 logits 还在，重新打包其他 blend 只需要几分钟。

如果这些中间文件已经删除，需要从原始数据重建：

- 重建或恢复稳定基线前置产物：约 1.5 到 3 小时。
- 构造上下文训练样本、训练、预测和打包：约 2 到 3 小时。
- Jittor/Ascend 编译和 I/O 波动：约 0.5 到 2 小时。

现实估计：4 到 8 小时。

## 9. 代码结构

现在只保留一个 shell 入口：

```bash
scripts/train_final.sh
```

它管理完整流程：

```bash
ACTION=all bash scripts/train_final.sh
```

也可以分阶段运行：

```bash
ACTION=build bash scripts/train_final.sh
ACTION=train bash scripts/train_final.sh
ACTION=predict bash scripts/train_final.sh
ACTION=package bash scripts/train_final.sh
ACTION=package-sweep bash scripts/train_final.sh
```

Python 主入口是：

```bash
python -m src.final_pipeline
```

核心文件：

- `src/final_pipeline.py`：最终命令入口和打包逻辑。
- `src/ranking_pipeline.py`：上下文增强排序模型的构造、训练、预测。
- `src/baseline_support.py`：读取稳定基线产物，提供困难候选和 MLP 工具。
- `src/data.py`：数据读写、提交 zip 生成、概率转换。
- `src/features.py`：图特征构造。
- `src/validation.py`：本地验证指标和 top1 变化统计。

## 10. 固化后的训练流程

1. 准备稳定基线：得到 dataset2 的规则/MLP logits，以及 dataset1 的稳定 CSV。
2. 构造困难候选：用稳定基线分数、热门目标、近期行为和转移关系生成训练样本。
3. 加入上下文特征：补充源对象行为序列、目标受众和 embedding 相似度。
4. 训练 3 个上下文排序 MLP：使用固定随机种子 `3101,3102,3103`。
5. 预测 dataset2：得到上下文增强 logits。
6. 打包最终结果：按 90/10 融合，输出 `result_final_blend_0p10.zip`。

## 11. 风险和注意事项

- 如果稳定基线前置产物缺失，最终结果不能直接复现。
- 如果只剩最终 zip，不能反推出其他 blend，因为 softmax 后原始 logits 信息已经丢失。
- `0.20`、`0.35`、`1.00` 这类版本更激进，可能涨，也可能明显跌。
- `0.10` 是已知线上最高分，应作为当前主线。
